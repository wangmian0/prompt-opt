# 数据报告

## 适用信号

- 输入含「统计/汇总/算一下/占比/趋势/环比同比/Top N/多少」等量化动词,要的是**数字或结论**,不是代码改动。
- 明确指向一批**已存在的数据**(日志文件、数据库表、埋点、CSV/表格),让你从中算出指标。
- 交付物是结论 / 数据表 / 图,不改线上行为、不动代码(区别于修 bug、加功能、重构)。
- 关注「数据说明了什么」,而非「某功能为什么坏了」(区别于排障)或「某方案该怎么选」(区别于调研)。

## 字段骨架

- **目标**:这份报告要回答的具体业务问题(是「回答 X」,不是「跑一下数据」);用户明说了下游决策才写决策,没说就只写问题,不编造。
- **上下文**:数据从哪来、给谁看(未指定则默认用户本人并标假设,不编造受众)、背景(已有分析、上一次结论、口径惯例)。
- **约束**:口径与边界 —— 明确不含哪些数据 / 不下哪些结论;信息不足处标「假设:xxx」不虚构;指涉环境中可自行发现的对象(报错 / 文件 / 数据 / 日志)先自查当前目录与上下文,确实找不到再问用户。
- **验收标准**:可验证 —— 关键数字给出计算口径或可复跑脚本能复算,结论与数字一一对应,产出能跑的查询 / 命令与可观察的表或图。
- **数据源与口径**:用哪份数据(表名 / 文件 / 字段),去重、过滤、关联规则怎么定;字段未知先自查表结构,再标假设。
- **指标定义**:每个指标怎么算(分子分母、单位、是否去重),避免同名不同义;口径是自定的就标假设。
- **时间范围**:统计的起止时间与粒度(日 / 周 / 月);「比以前」这类多义基期(环比 / 同比 / 历史均值)要把读法列出来标假设,不静默选一个。
- **输出形式**:结论先行还是明细表、要不要图,由「给谁看」决定详略。

## 示例

**口语输入**
> 帮我看下上个月订单退款情况,是不是比以前变多了?

**套骨架后的标准 prompt**
- **目标**:量化上个月退款情况,回答「是否较此前上升」。(原话未提下游决策,不编造)
- **上下文**:数据来自订单与退款记录。假设:受众为用户本人;退款存于 refunds 表,经 order_id 关联 orders(表名先自查库 / 文档确认)。
- **约束**:只统计已完成退款,不含申请中 / 已驳回;假设:「上个月」指上一个自然月;信息不足处标「假设:xxx」不虚构;表结构先自查再问用户。
- **验收标准**:给出可复跑的 SQL;关键数字可按口径复算;结论「变多 / 没变」与表中数字一致,并注明所选基期读法。
- **数据源与口径**:refunds 表(假设:金额与退款完成时间字段以实际表结构为准,先自查)。
- **指标定义**:「变多」可指退款笔数 / 退款金额 / 退款率,多种读法——假设:按退款率(退款订单数 / 同期总订单数)理解,读法需用户确认;仓库已有既定口径则以其为准。
- **时间范围**:「比以前」有多种读法——环比(vs 前一个自然月)/ 同比(vs 去年同月)/ 对比近 6 个月均值;假设:按环比理解;若退款有季节性,建议同时看同比,读法需用户确认。
- **输出形式**:结论先行(是否上升 + 幅度);可选建议(非原话要求):附周粒度趋势表看走势。
