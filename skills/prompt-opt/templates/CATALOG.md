# 场景模板索引

把口语化任务输入补齐成结构化标准 prompt 时,先按下表判定场景,再读对应模板套骨架。

| 场景 | 文件 | 一句判据 |
|------|------|----------|
| 排障 / 修 bug | debug-fix.md | 有具体的「坏了」:报错 / 崩溃 / 结果不对,要恢复正常 |
| 加功能 | feature.md | 要新增现在没有的能力,「用起来多了什么」 |
| 重构 | refactor.md | 嫌代码乱要整理,行为不变,常叮嘱「别改坏」 |
| 调研 / 研究 | research.md | 要答案 / 选型 / 可行性结论,本轮不动代码 |
| 代码审查 | review.md | 有明确对象(PR/diff/文件)要把关,交付发现清单不改代码 |
| 数据报告 | data-report.md | 对已存在的数据算数字 / 出结论(统计、占比、趋势) |

## 路由规则(输入同时命中多个场景时)

先按**主交付物**分流:要动手改代码 → 修 bug / 加功能 / 重构;不动代码 → 审查(发现清单)/ 调研(结论建议)/ 数据报告(数字)。再用下面的细则裁决;仍拿不准就列 top-2 候选让用户选,不硬套。

- **修 bug vs 代码审查**:指向一个具体故障(有报错 / 复现)→ 修 bug;对一段代码 / PR 做面上把关(「有没有坑」)→ 审查。「PR 里有处算错了,顺便看看还有没有别的坑」→ 审查(主交付是清单,那处错是清单第一条)。
- **重构 vs 代码审查**:要你动手整理 → 重构;要你列问题清单 → 审查。「帮我看看能不能整理」没说清要哪个 → 列两个候选问用户。
- **加功能 vs 修 bug**:现在完全没有的能力 → 加功能;已有能力在某些输入下出错(如中文乱码)→ 修 bug。
- **加功能 vs 调研**:「加个 X,用 A 还是 B 好?」——单条输入判不出要先选型还是直接落地,列两个候选问用户(先调研出选型结论 / 按假设选型直接加),不硬套。
- **调研 vs 数据报告**:答案来自资料 / 方案比较 → 调研;答案来自对自家数据算数 →数据报告。「查一下退款是不是变多了」→ 数据报告(「查」的对象是数据不是方案)。

## 模板格式约定

每个模板三节:适用信号(判定用)/ 字段骨架(基底四字段 目标 / 上下文 / 约束 / 验收标准 + 场景特有字段)/ 示例(口语输入 → 标准 prompt)。通用纪律:意图保真第一,不改意图、不扩范围;信息不足标「假设:xxx」不虚构,多义处把读法列出来不静默选;环境可自查对象先自查再问。
