# 排障 / 修 bug

## 适用信号

- 有明确的「坏了 / 不对」信号:报错、崩溃、异常码、结果算错、行为与预期不符。
- 贴了报错原文 / 堆栈 / 日志片段,或用「本来应该……结果却……」描述落差。
- 目标是让现有功能恢复正常,不是新增能力(区别于加功能)、不是无行为变化地改内部结构(区别于重构)。
- 指向一个具体故障根因,不是对整体质量做面上评估(区别于代码审查)。
- 常带复现或触发条件(「每次点 X 就……」「只有 Y 时才……」),而非开放式对比调研(区别于调研研究)。

## 字段骨架

- **目标**:要修好的是什么——哪个功能 / 行为恢复到什么正常状态才算修好。
- **上下文**:bug 落在哪个模块 / 文件 / 功能路径,牵涉的代码与系统范围。
- **约束**:修复边界(改动范围、不可动的部分、根因修复而非掩盖症状);信息不足处标「假设:xxx」不虚构;报错 / 文件 / 数据 / 日志等环境中可自查的对象,先查目录与上下文再问用户。
- **验收标准**:可验证的修好证据——写一个能复现该 bug 的测试,修复后跑通并贴测试输出(能跑的命令 / 可观察的产物);偶发 / 时序类 bug 无法稳定复现时,改为给出概率复现脚本或日志 / 监控证据,并说明为何不能确定性复现。
- **现象与报错原文**:一字不差的报错 / 堆栈 / 异常码,不转述、不改写。
- **复现步骤**:从什么状态、做哪几步触发,必然还是偶发。
- **预期 vs 实际**:本应发生什么,实际发生了什么。
- **已试过什么**:排查过哪些方向、改过什么、结果如何(避免重复无效路径)。
- **环境与版本**:OS / 运行时 / 依赖版本 / 分支 commit 等相关坐标。

## 示例

**口语输入**
> 我那个导出 csv 的功能又崩了,点一下就报错,你帮我看看。

**套骨架后的标准 prompt**
- **目标**:让 CSV 导出恢复正常——点导出不再崩溃报错(假设:导出的具体形态以现有实现为准,先自查确认)。
- **上下文**:CSV 导出功能。假设:代码在本仓库,可搜「导出 / csv / export」定位入口。
- **约束**:只修导出崩溃这一处,不顺手改其他功能;根因修复不掩盖症状。信息不足处标「假设」不虚构;报错与相关文件先在仓库自查再问我。
- **验收标准**:补一个复现「点击导出即崩溃」的测试,修复后该测试通过,贴测试输出。
- **现象与报错原文**:用户未提供。假设:需先复现并抓取真实报错 / 堆栈(不虚构报错内容)。
- **复现步骤**:点击导出按钮 → 崩溃(具体前置状态待复现时补全);「又崩了」提示疑似回归,先查近期相关改动。
- **预期 vs 实际**:预期导出成功(形态以实现为准);实际点击即崩溃报错。
- **已试过什么**:用户未说明。假设:暂无已试路径。
- **环境与版本**:未提供,先自查仓库(语言 / 依赖 / 分支)确认。
