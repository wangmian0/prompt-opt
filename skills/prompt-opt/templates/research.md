# 调研 / 研究

## 适用信号

- 用户要的是「答案 / 结论 / 建议」而非代码改动:问「该用 X 还是 Y」「能不能做到 Z」「业界怎么解决 W」,落点是判断,不是落地。
- 出现调研 / 研究 / 对比 / 选型 / 可行性 / 预研 /「查一下」「搞清楚」等词,面对的是开放问题,而非已知故障(区别于排障:没有坏掉要修的东西)。
- 需要汇集外部信息或横向比较多个方案再下判断,而不是在既有代码里直接动手(区别于加功能 / 重构:本轮不产出代码改动)。
- 关注「要不要做、怎么做才对」的方案层决策,产出喂给一次立项 / 选型 / 取舍(区别于代码审查:不是逐行挑已写代码的毛病)。
- 交付是结论 / 建议 / 报告,依据来自资料与代码阅读,而非对结构化数据做统计与可视化(区别于数据报告:不以数据表 / 图表为核心产物)。

## 字段骨架

- **目标**:这次调研要弄清的核心问题是什么,要拿到什么形态的结论(答案 / 建议 / 取舍),本轮是否只调研不动代码。
- **上下文**:为什么现在要查、已知什么、在什么项目 / 系统背景下做判断。
- **约束**:范围边界、深度 / 时间限制与不做什么;并含两条通用条款——信息不足处标「假设:xxx」不虚构;指涉环境中可自行发现的对象(报错 / 文件 / 数据 / 日志)先自查目录与上下文,再问用户。
- **验收标准**:本场景的可验证 = 每个问题都有明确答案、标注来源且可交叉核对;结论 + 行动清单在前(纯观点 / 可行性调研没有「能跑的命令」,来源可核对即为验收口径)。
- **要回答的问题清单**:把开放议题拆成 3–5 个可逐条回答的具体问题,避免「研究一下 X」式空泛。
- **谁看 + 解锁什么行动**:结论给谁看、看完要做的决策(选型 / 立项 / 放弃),据此定深度与呈现方式;未指定则默认用户本人并标假设,不编造受众与决策。
- **信息来源与可信度要求**:优先 / 允许哪些来源(官方文档 / 源码 / 业界实践 / 本仓库代码),是否要交叉核对并标出处。
- **结论格式**:结论先行(先给答案与建议),对比、依据、细节进附录,末尾能落成可执行的行动清单。

## 示例

**口语输入**
> 我们日志现在直接写文件太乱了,想搞个正经的日志方案,你研究下用啥好

**套骨架后的标准 prompt**
- **目标**:确定本项目日志体系的技术选型,产出「用哪个 + 为什么」的结论(假设:选型结论需附迁移代价评估);本轮只调研决策,不改代码。
- **上下文**:现状是日志直接写文件、比较乱(假设:痛点含难检索与无分级,以仓库实际为准先自查)。语言 / 运行环境与现有依赖先自查确认。
- **约束**:只在本项目现有语言与运行环境内选型;信息不足处标「假设:xxx」不虚构;涉及现有日志代码、配置、依赖清单先自查仓库目录与上下文,再问用户。
- **验收标准**:候选方案 2–3 个逐一给结论(是否推荐 + 理由 + 来源);推荐结论与迁移行动清单在前,对比细节进附录。
- **要回答的问题清单**:① 现在的写法具体卡在哪;② 有哪些候选(结构化日志 / 分级 / 落盘与检索);③ 各自迁移成本;④ 推荐哪个。
- **谁看 + 解锁什么行动**:假设:给用户本人看(原话未指定受众),看完决定是否立项替换日志方案。
- **信息来源与可信度要求**:优先本仓库现状 + 所用语言生态成熟库的官方文档,跨来源交叉核对并标出处。
- **结论格式**:先给推荐方案与一句话理由,候选对比表与迁移步骤进附录。
