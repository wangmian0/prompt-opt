# 代码审查

## 适用信号

- 输入指向「评价已有代码」而非改动它:「帮我 review 一下」「合并前把把关」「这段有没有问题」——交付物是判断 + 发现清单,不是可运行的代码改动(区别于修 bug / 加功能 / 重构:那三者都产出代码改动)。
- 已存在明确审查对象:一个 PR / commit / diff / 指定文件范围摆在那里,让你点评它,而不是让你生成或修改它。
- 常带关注维度或担忧词:「有没有坑」「安全吗」「会不会有 bug」「写得干不干净」——期待你挑毛病而非确认没问题。
- 针对本仓库既有代码的质量把关(区别于调研研究:后者探索外部知识 / 方案,不落到具体某段代码的对错)。

## 字段骨架

- **目标**:这次审查要判定什么(能否合并 / 找出某类问题 / 整体质量把关),明确交付发现而非代码改动。
- **上下文**:被审代码的来龙去脉(哪个 PR / 分支 / 需求,这堆改动想解决什么问题)。
- **约束**:审查规则(如只交付发现不改代码、是否审风格)——范围圈定写在「审查对象」,不在此重复;信息不足处标「假设:xxx」不虚构;指涉环境中可自行发现的对象(报错 / 文件 / 数据 / 日志)先自查目录与上下文再问用户。
- **验收标准**:可验证的交付形态——发现逐条列出,每条带严重度与证据(文件:行号),佐以能跑的复现命令 / 可观察产物;无发现也要说明验证了哪些维度、覆盖了哪些文件。
- **审查对象**:范围界定到 diff / PR / commit / 具体文件,而非「整个项目随便看看」。
- **关注维度**:正确性 / 安全 / 性能 / 简洁 / 风格,可多选并排出优先级。
- **证伪导向**:目标是找 bug、反例、边界与漏网场景,逐个 caller 与边界主动质疑,不是确认「没问题」。

## 示例

**口语输入**
> 这个 PR 加了个用户登录接口,合并前你帮我看看有没有坑,尤其是安全方面。

**套骨架后的标准 prompt**
- **目标**:对该登录接口 PR 做合并前审查,交付发现清单(假设:并附「能否合并」的建议),不直接改代码。
- **上下文**:PR 新增用户登录接口;需先自查该 PR 改动了哪些文件再动手。
- **约束**:只交付发现不改代码;信息不足处标「假设」不虚构;报错 / 依赖 / 配置等可自查对象先看目录与上下文再问。
- **验收标准**:发现逐条列出,每条标严重度(高 / 中 / 低)+ 证据(文件:行号)+ 复现或触发说明;无高危发现也说明验证了哪些安全维度、覆盖了哪些文件。
- **审查对象**:本 PR 的 diff(登录接口相关改动文件;假设:不含既有登录逻辑与未改动文件)。
- **关注维度**:安全为主(假设:细化为认证、凭据存储、注入、越权),兼顾正确性。
- **证伪导向**:针对每个入口参数与错误分支找绕过与边界,不预设实现正确。
